## 목표 (Goal)

제보는 **익명**(서버 세션 없음, 0002 결정 7)이라 제보자가 자기 제보의 진행 상황을 다시 볼 방법이 없다. 블루프린트의 PO★ 항목(유저 9명 요청): 제보자가 **접수번호로 자기 제보 상태를 추적**할 수 있어야 한다.

이 스펙은 (1) report 에 사람이 읽고 적을 수 있는 **공개 접수번호**(`VT-YYYY-MMDD-NNNN`)를 부여하고, (2) 제보 생성 응답·완료 화면에 그 번호를 노출하고, (3) **인증 없이** 접수번호로 단건 상태를 조회하는 **공개 상태조회 API/화면**(민감정보 비노출, 상태 타임라인만)을 정의하며, (4) 제보자가 자기 접수번호들을 로컬에 보관해 보는 **"내 제보 목록"** 화면을 정의한다.

## 비목표 (Non-goals)

- 서버 측 제보자 계정/세션/로그인 — 익명성 유지가 결정(아래 결정 5). "내 제보"는 **클라이언트 localStorage** 기반.
- 제보 내용·첨부·검증 근거·reviewer 신원의 공개 — 상태조회는 **타임라인 단계만** 반환. 상세 본문은 verified 후 0005 공개 경로로만.
- 접수번호로 제보 수정/삭제 — 추적(읽기)만. 수정은 관리자(0004) 권한.
- 이메일/SMS 알림(상태 변경 푸시) — 후속. 본 스펙은 pull 방식 조회만.
- 기존 0001~0012 의 동작 변경(회귀). report.id(uuid)는 그대로 유지하고 접수번호는 **추가** 식별자.

## 사용자 흐름 (User flow)

### A. 제보 생성 시 접수번호 발급
1. `POST /api/reports`(0002) 성공 → report 행에 접수번호 부여 → 응답 본문에 `trackingNumber`(예: `VT-2026-0615-0042`) 포함(기존 `id`/`status` 와 함께).
2. 입력 마법사(0003) 완료 화면에 접수번호를 **크게 표시** + "복사" + "이 번호로 상태를 조회할 수 있습니다" 안내. 완료 시 접수번호를 localStorage("내 제보")에 적재.

### B. 공개 상태조회 (인증 없음)
3. web 상태조회 화면: 접수번호 입력 → `GET /api/track/:trackingNumber` 호출.
4. 존재하면 **상태 타임라인**(접수됨 → 검수중 → 검증완료 → 공개)을 단계별 완료/현재/미도달로 표시. verified+공개면 0005 상세로 가는 링크 노출.
5. 없거나 형식 오류면 "해당 접수번호의 제보를 찾을 수 없습니다"(존재 여부 외 정보 누설 없음).

### C. 내 제보 목록
6. "내 제보" 화면: localStorage 에 보관된 접수번호 목록을 표시(번호 + 마지막 조회 상태). 각 항목 클릭 → 상태조회(B) 재호출로 최신 상태 갱신. 번호 직접 추가/삭제 가능.

## 수용 기준 (Acceptance criteria — 테스트 가능하게)

### 접수번호 발급 (서버)
- [ ] `POST /api/reports` 성공 응답에 `trackingNumber` 가 포함되고, 형식 정규식 `^VT-\d{4}-\d{4}-\d{4}$` 에 매칭된다.
- [ ] 같은 날 연속 생성된 두 제보의 `trackingNumber` 가 **서로 다르다**(충돌 없음). 같은 날 NNNN 시퀀스가 증가한다.
- [ ] 접수번호는 report 행에 저장되고 **유니크 제약**이 있다. 동일 번호 재발급 시도는 DB 레벨에서 거부된다.
- [ ] `trackingNumber` 의 날짜 부분은 서버 시각(collected_at) 기준이다(클라이언트 시각 불신, 0002 결정과 일관).

### 공개 상태조회 (서버)
- [ ] `GET /api/track/:trackingNumber` 는 **인증 없이** 호출 가능하고, 존재하면 `{ trackingNumber, timeline: [...단계], currentStage }` 를 반환한다.
- [ ] 응답에 제보 **본문·첨부·submitter 해시·sources·reviewer 신원·내부 status 원문 외 민감정보가 없다**(직렬화 계약 테스트로 단언). 노출은 접수번호·단계 라벨·각 단계 도달 시각(있으면)으로 한정.
- [ ] 존재하지 않는/형식 불일치 접수번호는 **404**(미검증 제보 존재 누설 방지 — verified 여부와 무관하게 단계만 표기하되, 행 자체가 없으면 404).
- [ ] verified=true 인 제보의 상태조회 응답에는 0005 공개 상세로 가는 `publicUrl`(또는 id)이 포함된다. verified 가 아니면 publicUrl 은 null/미포함.
- [ ] 상태조회에도 IP rate limit 이 적용된다(번호 무차별 조회 방지) — 초과 시 **429**.

### 상태 타임라인 매핑
- [ ] report.status 내부 enum → 공개 단계 매핑이 결정 3 표대로 동작한다(예: submitted/pending→접수됨, reviewing→검수중, confirmed/verified→검증완료, 공개노출 가능→공개). 매핑 외 미지 status 는 "접수됨"으로 폴백.

### web 화면
- [ ] 완료 화면(0003)에 접수번호가 렌더되고, "복사" 동작이 클립보드에 접수번호를 넣는다(모킹). 완료 시 localStorage "내 제보"에 추가된다.
- [ ] 상태조회 화면에서 유효 번호 입력 → API 응답의 타임라인 단계가 완료/현재 표시로 렌더된다. 없는 번호 → "찾을 수 없습니다" 메시지.
- [ ] "내 제보" 화면이 localStorage 의 번호 목록을 렌더하고, 항목 클릭 시 상태조회 API 를 호출한다(모킹). 빈 목록일 때 안내 문구.

## 설계 메모 (Design notes)

### 결정 / 근거 (자율 진행 — 보수적 기본값)

1. **접수번호 형식 = `VT-YYYY-MMDD-NNNN`**(요청 명시). `VT`=Votatis 접두. YYYY-MMDD=발급일(서버 시각). NNNN=그날의 4자리 일련번호(0001부터, 9999 초과 시 5자리 확장은 후속 — MVP 일일 9999건 가정 충분). 사람이 읽고 받아쓰기 쉬운 대시 구분.
2. **충돌 없는 생성 = DB 유니크 + 트랜잭션 시퀀스**. NNNN 은 "해당 날짜 prefix 의 기존 최대값+1" 을 같은 트랜잭션에서 계산하거나, 일별 카운터 테이블/시퀀스로 원자적 증가. 유니크 제약을 안전망으로 두고, 충돌 시 재시도. (랜덤 4자리는 충돌·추측 위험 → 순증 채택. 단 추측 방지는 결정 4 참고.)
3. **상태 타임라인 매핑**(내부 status → 공개 단계, 4단계 고정):
   | 공개 단계 | 매핑되는 내부 status |
   |-----------|----------------------|
   | 접수됨    | submitted, pending, unverified |
   | 검수중    | reviewing |
   | 검증완료  | confirmed, (verification.verified=true) |
   | 공개      | verified=true 이며 0005 공개 노출 대상 |
   미지/미래 status 는 "접수됨"으로 폴백(누설보다 보수적). 매핑 표는 서버 단일 상수로 두어 0001 status enum 변경 시 한 곳만 갱신.
4. **추측 방지(보안)**: 순증 NNNN 은 추측이 쉬우므로 — 상태조회는 (a) 민감정보 0 노출(타임라인만), (b) rate limit, (c) 행 없으면 404 로 방어. 추가 난수 salt 토큰은 "받아쓰기 쉬움" 요구와 상충 → MVP 미채택, 노출 최소화로 대응(문서화된 트레이드오프).
5. **익명 유지 = 서버 세션 없음**. "내 제보"는 **클라이언트 localStorage** 에 접수번호 배열을 보관(서버는 제보자-번호 매핑을 식별 저장하지 않음). 기기 변경 시 번호를 직접 입력하면 조회 가능. 이는 0002 결정 7(익명 해시)·프라이버시 방향과 일관.

### 데이터 무결성
- trackingNumber 는 **불변 식별자**(발급 후 변경 금지). report 수정 시에도 그대로 유지 → report_history 스냅샷에 포함되어 보존(0001 패턴, 별도 처리 불필요).
- 날짜 부분은 collected_at(서버 시각) 파생 — 사후 재계산 없이 발급 시 고정.

### 0002/0003/0005 계약 정렬
- 0002 `POST /api/reports` 응답에 `trackingNumber` 필드만 **추가**(기존 `id`/`status` 불변 → 회귀 0).
- 신규 라우트 `GET /api/track/:trackingNumber` 는 공개 네임스페이스(admin 아님). 직렬화는 0005 공개 상세와 별개의 최소 페이로드.
- web 라우팅에 `/track`(상태조회)·`/my`(내 제보) 추가. 완료 화면은 0003 의 마지막 단계에 접수번호 블록 삽입.

### 회귀 (절대 조건)
- 기존 0001~0012 테스트 전부 그대로 통과해야 한다. report.id(uuid) 경로·공개 조회 계약·민감 필드 비노출 불변.
