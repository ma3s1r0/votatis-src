## 목표 (Goal)

현재 `apps/web` 은 기능은 동작하지만 디자인이 없다(`index.css` 는 system-ui 폰트·기본 마진뿐, 컴포넌트는 무스타일). Figma 디자인 정본([votatis-figma], https://www.figma.com/design/VxSqEMKJRYnYZGb0QoZN5x/Votatis)을 Figma MCP(원격, 인증 완료)로 읽어 **디자인 토큰(색·타이포·간격)**을 도입하고, **핵심 화면 1~2개**가 그 토큰 기반으로 렌더되게 한다.

디자인은 주관·범위가 커서 폭주하기 쉽다. 이 스펙은 의도적으로 **MVP 스코프로 한정**한다: 토큰 도입 + 핵심 화면(우선 랜딩·공개 아카이브 또는 제보 마법사 중 1~2개)이 토큰 기반. 픽셀 완벽·전체 화면 적용은 후속.

## 비목표 (Non-goals)

- 모든 화면(랜딩·제보 마법사·공개 아카이브·관리 콘솔 전부)의 픽셀 단위 일치 — 후속. 본 스펙은 토큰 + 핵심 1~2화면.
- 디자인 시스템/컴포넌트 라이브러리 풀 구축(Storybook·전체 컴포넌트 카탈로그) — 후속.
- 다크모드·테마 전환 — 토큰 구조는 변수 기반으로 두되, 다크 테마 값 세트는 후속.
- 코드 → Figma 역방향 동기화, Code Connect 매핑 — 본 스펙은 디자인 → 코드 단방향.
- 애니메이션·마이크로 인터랙션 — 후속.
- 새 기능/플로우 추가 — 본 스펙은 기존 0003/0005 화면의 **스타일**만 입힌다. DOM 구조·동작 변경 최소화(기존 테스트 깨지 않게).

## 사용자 흐름 (User flow)

(기능 흐름은 0003/0005 그대로. 본 스펙은 시각 표현만 바꾼다.)
1. 사용자가 랜딩/아카이브/제보 화면에 진입 → 디자인 토큰 기반의 일관된 색·타이포·간격으로 렌더.
2. 톤은 0003/0005 결정대로 "객관적 데이터 서비스" — 디자인도 단정/선동 톤이 아닌 중립·정보 중심.

## 수용 기준 (Acceptance criteria — 테스트 가능하게)

### 디자인 토큰
- [ ] Figma에서 읽은 색·타이포·간격이 **CSS 변수(또는 동등한 토큰 모듈)**로 정의된다(예: `--color-*`, `--font-*`, `--space-*`). 토큰 출처는 단일 파일.
- [ ] 토큰이 임의 하드코딩이 아니라 Figma 디자인 값에서 유래함을 주석/문서로 추적 가능(어느 프레임·스타일에서 왔는지 메모).

### 핵심 화면(1~2개) 토큰 기반 렌더
- [ ] 선정된 핵심 화면(결정 2)의 주요 요소(배경·텍스트·버튼·간격)가 **토큰 변수를 참조**한다(스타일에 raw hex/px 직접값 대신 var() 사용 — 핵심 화면 한정으로 단언 가능).
- [ ] 해당 화면이 렌더되고, 기존 기능 테스트(0003/0005의 RTL 테스트)가 **여전히 통과**한다(스타일 적용이 동작·접근성 라벨·카피를 깨지 않음).
- [ ] 0003/0005에서 단언한 "관찰·기록 톤 / 데이터·검증 중심 톤" 고정 카피가 유지된다(디자인 적용 후에도 카피 단언 통과).

### 회귀 안전
- [ ] `pnpm -r typecheck && pnpm -r build && pnpm -r test` 가 통과한다(스타일 변경이 빌드/테스트를 깨지 않음).
- [ ] 토큰 변수 1개 변경 시 핵심 화면의 해당 시각 속성이 한 곳에서 바뀐다(토큰이 실제로 single source of truth임을 보이는 최소 확인 — 스냅샷 또는 computed style 단언 택1).

## 테스트 계획 (TDD — Red 먼저)

- 디자인은 시각 주관이 강해 "픽셀 일치"는 테스트하지 않는다. 대신 **구조적·회귀적**으로 검증:
  - `apps/web`: 핵심 화면 컴포넌트가 토큰 클래스/변수를 사용하는지(클래스명 또는 인라인 var 존재 단언), 기존 RTL 테스트(0003 `ReportWizard.*`, 0005 `archive.*`) 그린 유지.
  - 토큰 모듈 단위: 정의된 토큰 키 집합이 존재하는지(예: 필수 색/타이포 키 누락 시 실패).
  - 카피 단언(0003/0005) 재실행으로 톤 회귀 방지.
- 시각 회귀(스크린샷 diff)는 도입 시 비용·플레이크 큼 → MVP 비목표. 필요 시 후속에 Playwright 스냅샷.

## 설계 메모 (Design notes)

### 결정 / 근거 (자율 진행 — 보수적·범위 한정)

1. **토큰 = CSS 변수**(`:root` 의 `--color-*`/`--font-*`/`--space-*`). 근거: Vite+React 기본에 무의존, 런타임 비용 0, 다크모드 확장 여지. CSS-in-JS·Tailwind 도입은 큰 결정이라 MVP에서 보류(필요 판단 시 별도 스펙).
2. **핵심 화면 = 공개 아카이브 리스트(0005) + 제보 마법사(0003) 중 우선 1개, 가능하면 2개**. 근거: 둘 다 대표 사용자 흐름(탐색·기여)이고 카피 톤 단언이 이미 있어 회귀 안전망이 존재. 랜딩은 신규 페이지라 범위 증가 → 토큰만 깔고 후속. 최종 선정은 Figma에서 어떤 프레임이 가장 완성도 있는지 보고 1줄 합의.
3. **Figma 접근 = 원격 MCP**(get_design_context/get_screenshot/get_metadata). node-id 파싱 후 프레임별 토큰·레이아웃 추출. `/figma-use` 등 스킬은 코드→디자인(use_figma) 시 필수지만 본 스펙은 디자인→코드라 읽기 도구 위주.
4. **DOM 변경 최소화**: 기존 0003/0005 컴포넌트의 마크업·role·라벨·data-testid 를 보존하고 className/스타일만 추가. 근거: 기존 RTL 테스트(api 89 + web 50대)를 회귀 없이 유지. 구조 개편이 필요하면 별도 스펙으로 분리.
5. **스코프 게이트**: 토큰 + 핵심 1~2화면을 넘는 작업(전 화면 적용·컴포넌트 라이브러리·반응형 전 break point)은 본 스펙에서 멈추고 후속 스펙으로. "완벽한 디자인"이 아니라 "토큰 도입 + 핵심 화면 토큰 기반 렌더"가 완료 정의.

### 무결성 / 톤
- 디자인이 0003/0005의 객관·중립 톤(부정선거 단정 아님)을 시각적으로도 해치지 않게: 경고/위험을 과장하는 색·카피 배치 지양. 검증 상태(verified/unverified)는 중립 배지로.

### 리스크
- Figma 디자인이 미완성/일부 프레임만 있을 수 있음 → 그 경우 존재하는 프레임에서 토큰만 추출하고 화면 적용은 가용 범위로 축소(가정: 토큰 추출은 최소 1프레임이면 가능). 디자인 부재 시 사용자에게 어떤 프레임을 기준할지 확인.
