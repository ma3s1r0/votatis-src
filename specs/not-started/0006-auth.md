## 목표 (Goal)

검증된 인원만 라벨링·검증·수정을 할 수 있도록 **관리자 인증·권한 체계**를 만든다. 루트 관리자가 계정을 발급하고, 발급받은 사람은 **초대 링크에서 본인이 직접 비밀번호를 설정**한 뒤 로그인한다. 세션이 유지되며, **검증된(active) 관리자만** 데이터 수정 엔드포인트에 접근할 수 있다.

이 스펙은 0004(라벨링·검증 콘솔)의 **선행 조건**이다. 0004의 모든 쓰기 동작은 여기서 정의하는 인증·권한 미들웨어를 통과해야 한다.

## 비목표 (Non-goals)

- 일반 시민(제보자) 회원가입/로그인 — MVP에서 제보는 **익명 해시**로만(0001/0002). 시민 계정은 없음.
- 소셜 로그인(OAuth/구글 등), SSO, MFA/2FA — 후속.
- 세분화된 RBAC(역할별 권한 매트릭스). MVP는 **root / reviewer 2단계**만.
- 비밀번호 재설정 이메일 플로우 전체(분실 복구) — MVP는 root가 초대 재발급으로 대체. (SES 연동은 후속)
- 계정 잠금/감사 대시보드 UI.

## 사용자 흐름 (User flow)

1. **루트 부트스트랩**: 최초 root 관리자는 시드(환경변수/마이그레이션 시드 스크립트)로 1명 생성. UI에서 가입 불가.
2. **초대 발급** (root): 콘솔에서 이메일 입력 → `POST /api/admin/invites` → `admin_user`(status=invited) + `admin_invite`(일회용 토큰, 만료) 생성. 토큰이 담긴 초대 URL 반환.
3. **비밀번호 설정** (피초대자): 초대 URL 접속 → 토큰 검증 → 본인 비밀번호 입력 → `POST /api/admin/invites/:token/accept` → 비밀번호 해시 저장, status=active, 토큰 소비(재사용 불가).
4. **로그인**: `POST /api/admin/login` (email+password) → 검증 성공 시 서버 세션 생성, **httpOnly 쿠키** 발급.
5. **세션 유지**: 이후 요청은 쿠키의 세션 ID로 인증. `GET /api/admin/me` 로 현재 사용자 조회.
6. **권한 게이트**: 보호된 엔드포인트(0004 쓰기 등)는 `requireReviewer` 미들웨어 통과 필요 — 세션 유효 + user.status=active.
7. **로그아웃**: `POST /api/admin/logout` → 서버 세션 폐기, 쿠키 만료.

## 수용 기준 (Acceptance criteria — 테스트 가능하게)

- [ ] root 시드 계정이 존재하고, UI/공개 API로는 신규 계정을 self-가입할 수 없다(가입 엔드포인트 없음).
- [ ] root가 `POST /api/admin/invites` 호출 시 `admin_user(status=invited)` + 일회용 토큰이 생성되고 초대 URL이 반환된다.
- [ ] 비-root(reviewer) 또는 비인증 사용자가 초대 발급을 시도하면 **403**(인증 없으면 401).
- [ ] 유효한 초대 토큰 + 비밀번호로 accept 하면 status=active가 되고, 비밀번호는 **평문이 아닌 해시**로 저장된다.
- [ ] 이미 소비된/만료된/존재하지 않는 토큰으로 accept 하면 **거부(400/410)**, 계정 status는 변하지 않는다.
- [ ] 올바른 email+password 로그인 시 **httpOnly·Secure·SameSite=Lax 쿠키**가 세팅되고, `GET /api/admin/me` 가 해당 사용자를 반환한다.
- [ ] 틀린 비밀번호/없는 이메일 로그인은 **401**, 메시지는 둘을 구분하지 않는다(계정 존재 여부 누설 금지).
- [ ] 세션 쿠키 없이 보호된 엔드포인트 접근 → **401**. invited(아직 비번 미설정) 사용자 세션으로 접근 시도 → **403**.
- [ ] 로그아웃 후 동일 세션 쿠키로 보호된 엔드포인트 접근 → **401**(세션 폐기 확인).
- [ ] 로그인 시도에 **IP+계정 단위 rate limit** 적용(0001식 DB 기반). 임계 초과 시 **429**.

## 설계 메모 (Design notes)

### 결정 / 근거 (자율 진행 — 보안 민감 기본값은 보수적으로)

1. **세션 방식 = 서버 세션 + httpOnly 쿠키**(JWT-in-localStorage 아님). 근거: 서버측 즉시 폐기(로그아웃·권한 회수) 가능, XSS로 토큰 탈취 위험 낮음. 쿠키 속성: `HttpOnly; Secure; SameSite=Lax; Path=/`.
2. **비밀번호 해시 = argon2id**(없으면 bcrypt cost≥12). 평문/단방향 약한 해시 금지. 해시는 `admin_user.password_hash`에만 저장.
3. **초대 토큰** = 충분한 엔트로피(≥32바이트 random)의 일회용 토큰, **해시하여 DB 저장**(원문은 URL에만). 만료 기본 **72시간**. accept 시 소비(used_at 기록) → 재사용 거부.
4. **권한 모델 = 2단계**: `role ∈ {root, reviewer}`, `status ∈ {invited, active, disabled}`. `requireReviewer` = 세션 유효 ∧ status=active. `requireRoot` = 추가로 role=root. 0004 쓰기는 `requireReviewer`.
5. **rate limit = DB 기반**(0001 패턴 재사용). 로그인 실패 IP+email 키, 기본 윈도/임계 보수적(예: 5분당 10회). 외부 캡차는 후속 옵션.
6. **CSRF**: SameSite=Lax + 상태변경은 비-GET. 별도 CSRF 토큰은 콘솔이 별도 오리진/서브도메인일 때 추가 검토(후속 메모). MVP 동일 오리진 가정.
7. **세션 만료**: idle 기본 12h, absolute 7d. 서버 세션 테이블에 `expires_at`.

### 스키마 (0001 위에 추가 — verification.reviewer FK 연결)

- `admin_user(id, email unique, role, status, password_hash nullable, created_at)`
- `admin_invite(id, admin_user_id, token_hash, expires_at, used_at nullable)`
- `admin_session(id, admin_user_id, expires_at, created_at)` — 세션 ID는 쿠키에 (해시 저장 권장).
- 0001 `verification.reviewer` 는 `admin_user.id` 를 참조하도록 연결(0004에서 사용).
