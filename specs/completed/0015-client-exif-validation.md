## 목표 (Goal)

디자인 화면 11("원본 사진이 아닙니다" 차단 화면)이 전제하는 클라이언트 1차 검증을 정의한다. 제보폼에서 파일을 첨부할 때 **브라우저에서 EXIF 를 추출하고 MIME 을 이중검증**한다. EXIF 의 촬영시각·기기 정보가 없으면(스크린샷·캡처·재가공 의심) "원본 사진이 아닙니다" **차단 화면**을 띄워 업로드를 막고, EXIF 설명과 원본 재업로드 방법을 안내한다. "직접 촬영한 원본만" 문구를 노출한다.

서버 finalize(0002)의 무결성 검증(sha256·객체 존재)은 그대로 유지된다. 본 스펙은 **클라이언트 1차 차단**으로, 명백한 비원본을 업로드 전에 거른다.

## 비목표 (Non-goals)

- 서버 측 EXIF 강제/차단 — 0002 finalize 그대로. 서버는 exif jsonb 보관만(0001). 클라 차단은 UX·대역폭 절감용 1차 게이트이며 보안 경계가 아님(우회 가능 명시).
- EXIF 위변조 탐지·딥페이크 판별 — 비목표. "EXIF 존재 여부" 수준의 단순 검사.
- 동영상 EXIF/메타데이터 검증 — 본 스펙은 이미지(jpeg/png/webp)만. PDF 는 EXIF 검사 제외(결정 4).
- 위치정보(GPS) 노출 정책 — 별개 관심사. 본 스펙은 촬영시각·기기 **존재**만 본다.
- 서버 API 변경 — 본 스펙은 web 전용. 0002 계약 불변.

## 사용자 흐름 (User flow)

1. 제보폼(0003) 첨부 단계 → 사용자가 파일 선택.
2. 브라우저가 (a) 파일 매직넘버로 **실제 MIME** 판별 → 확장자/신고 MIME 과 대조, (b) 이미지면 **EXIF 파싱**.
3. EXIF 에 촬영시각(DateTimeOriginal 등) **또는** 기기(Make/Model) 정보가 **둘 다 없으면** → 차단 화면(화면 11): "원본 사진이 아닙니다" + EXIF 가 무엇인지 설명 + "직접 촬영한 원본만 업로드해 주세요" + 다른 파일 선택 버튼. 해당 파일은 첨부 목록에 추가되지 않는다.
4. MIME 이중검증 실패(확장자와 실제 내용 불일치)도 차단.
5. 검증 통과 파일만 첨부 목록에 추가되고, 이후 0002 create/finalize 플로우 진행.

## 수용 기준 (Acceptance criteria — 테스트 가능하게)

- [ ] EXIF 에 촬영시각·기기 정보가 모두 없는 이미지(예: 스크린샷)를 선택하면 차단 화면이 렌더되고, 그 파일은 첨부 목록에 추가되지 **않으며**, 0002 attachments/create 호출이 **발생하지 않는다**(모킹으로 미호출 단언).
- [ ] EXIF 에 촬영시각 또는 기기 정보가 있는 이미지는 통과해 첨부 목록에 추가된다.
- [ ] 실제 내용 MIME 과 신고 MIME(확장자)이 불일치하면(예: `.jpg` 인데 실제 PDF) 차단된다.
- [ ] 차단 화면에 "원본 사진이 아닙니다", EXIF 설명 문구, "직접 촬영한 원본" 안내, 다른 파일 선택 액션이 모두 렌더된다.
- [ ] 차단 후 다른(유효) 파일을 선택하면 정상 첨부된다(차단 상태가 영구 고정되지 않음).
- [ ] PDF 첨부는 EXIF 검사 대상에서 제외되어 차단되지 않는다(MIME 검증은 적용).
- [ ] EXIF 파싱이 실패하거나 라이브러리가 메타를 못 읽는 경우의 폴백 동작이 결정 3 대로다(차단 또는 경고 — 일관).

## 설계 메모 (Design notes)

### 결정 / 근거 (자율 진행 — 보수적 기본값)

1. **검증 = 클라이언트 1차 게이트, 보안 경계 아님**. 사용자가 코드를 우회하거나 EXIF 를 조작할 수 있음을 인정. 목적은 (a) 명백한 스크린샷/캡처를 업로드 전에 거르고 (b) 사용자에게 "원본만" 규범을 학습시키는 것. 서버 무결성(0002 finalize)이 권위.
2. **"원본" 판정 기준 = 촬영시각(DateTimeOriginal) OR 기기(Make/Model) 중 하나라도 존재**. AND 가 아닌 OR 인 이유: 일부 정상 카메라/기기가 한쪽만 기록. 둘 다 없을 때만 차단(거짓양성 최소화). 보수적으로 거짓양성보다 누락을 허용.
3. **EXIF 파싱 실패 시 폴백 = 경고 후 진행 허용**(차단 아님). 근거: 파서 한계로 정상 원본까지 막으면 제보 위축. "메타를 확인하지 못했습니다 — 직접 촬영한 원본인지 확인해 주세요" 경고만. (보안 경계가 아니므로 fail-open 이 UX 상 합리적. 서버가 최종 게이트.)
4. **EXIF 검사 대상 = 이미지(jpeg/png/webp)만**. PNG/WebP 도 EXIF 가 없는 경우가 흔해 거짓양성 위험 → 우선 jpeg 에 강검사, png/webp 는 결정 2 기준 적용하되 폴백(결정 3) 관대. PDF 는 EXIF 비대상(MIME 검증만).
5. **MIME 이중검증 = 매직넘버(파일 시그니처) vs 신고 MIME/확장자 대조**. FileReader 로 앞부분 바이트를 읽어 jpeg(FFD8)/png(89504E47)/webp(RIFF…WEBP)/pdf(%PDF) 판별. 불일치 시 차단(위장 파일 방지).
6. **라이브러리 선택**은 구현 착수 시 결정(예: exifr 등 경량). 본 스펙은 순수 판정 함수 인터페이스만 고정해 테스트가 라이브러리에 비의존.
