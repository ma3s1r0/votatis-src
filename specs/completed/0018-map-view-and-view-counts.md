## 목표 (Goal)

디자인이 전제하는 두 가지 공개 기능: (1) 전국 제보 분포를 보여주는 **지도 뷰** — 시도별로 검증 통과 제보를 핀으로 집계하고 상태별 색으로 구분, 범례에 건수 표시. (2) 공개 상세의 **조회수** 집계 — 상세를 볼 때 증가하고 화면에 표시.

지도는 실 지도 타일/지오코딩 없이 **시도 중심 좌표 정적 매핑 + 간단한 SVG/플롯**으로 MVP 를 만든다(인프라 무거운 부분 비목표).

## 비목표 (Non-goals)

- **실 지도 타일·벡터맵·지오코딩 서비스 연동**(Mapbox/구글맵/주소→좌표) — 비목표. 시도(17개) 중심 좌표를 정적 상수로 매핑하고 SVG/플롯으로 핀 배치.
- 시군구·읍면동 단위 핀 — MVP 는 시도 단위 집계만. 하위 단위는 후속.
- 클러스터링·줌·팬 인터랙션 고도화 — 단순 정적 분포도. 핀 클릭→해당 시도 필터(0005) 연결 정도.
- 정밀 조회수(유니크 방문자·중복 제거·봇 필터) — MVP 는 상세 조회 시 단순 증가(결정 4). 어뷰징 방지는 후속.
- 실시간 집계·캐시 무효화 — 조회수는 즉시 증가, 지도 집계는 요청 시 계산(또는 짧은 캐시).

## 사용자 흐름 (User flow)

### A. 지도 뷰
1. 공개 아카이브(0005)에 "지도" 탭/뷰 → `GET /api/reports/map-stats`(또는 기존 목록에 집계 옵션) 호출.
2. 응답: 시도별 `{ sido, total, byStatus: { verified, reviewing, unverified } }` 배열.
3. web: 시도 중심 좌표 정적 매핑으로 SVG 위에 핀 배치. 핀 색 = 상태(검증됨 초록 / 검증중 주황 / 미검증 빨강), 크기/배지 = 건수. 범례에 상태별 색·총건수.
4. 핀 클릭 → 해당 sido 로 0005 목록 필터(기존 sido 필터 재사용).

### B. 조회수
5. 공개 상세(0005 / 0002 `GET /api/reports/:id`) 진입 → 서버가 조회수 +1.
6. 응답·화면에 현재 조회수 표시.

## 수용 기준 (Acceptance criteria — 테스트 가능하게)

### 지도 집계 (서버)
- [ ] `GET /api/reports/map-stats` 가 시도별 집계 배열을 반환한다. 각 항목에 `sido`, `total`, 상태별 건수가 포함된다.
- [ ] 지도 집계는 **공개 가시성 규칙을 따른다**(0002) — 미검증 제보의 본문은 누설하지 않되, 집계 카운트 정책은 결정 2 대로(검증됨만 vs 상태별 모두) 일관 동작한다.
- [ ] sido 가 null/미지인 제보는 "미지정" 버킷으로 분리되거나 제외(결정 3)되며, 합계가 깨지지 않는다.

### 조회수 (서버)
- [ ] `GET /api/reports/:id`(공개 상세) 호출 시 해당 report 의 조회수가 1 증가하고, 응답에 `viewCount` 가 포함된다.
- [ ] verified 가 아닌(404 되는) 제보 조회는 조회수를 증가시키지 않는다(존재 누설·집계 오염 방지).
- [ ] 조회수 증가는 동시 요청에서도 유실 없이 누적된다(원자적 증가 — 단위/통합 테스트로 단언).

### web
- [ ] 지도 뷰가 시도별 핀을 렌더하고, 핀 색이 상태별(초록/주황/빨강)로 매핑된다. 범례에 상태별 색·건수가 표시된다.
- [ ] 핀 클릭 시 해당 sido 로 0005 목록 필터가 적용된다(쿼리 호출 모킹 단언).
- [ ] 공개 상세 화면에 조회수가 렌더되고, 진입 시 API 의 viewCount 가 표시된다.
- [ ] 좌표 미매핑 시도(상수에 없는 sido 표기)는 핀 누락이 아니라 "미지정/목록 외" 로 안전 처리(렌더 깨짐 없음).

## 설계 메모 (Design notes)

### 결정 / 근거 (자율 진행 — 보수적 기본값)

1. **지도 = 시도 중심 좌표 정적 상수 + SVG/플롯**(요청 명시 비목표 경계). 17개 시도(서울/부산/…/제주)의 대략적 중심 위경도(또는 SVG viewBox 좌표)를 코드 상수로 둔다. 실 타일/지오코딩 없음. 좌표 상수는 단일 출처(서버 또는 web 공유), 정밀도는 분포 표시에 충분한 수준.
2. **지도 집계 카운트 정책 = 상태별 모두 집계하되 공개 안전 범위로**. 색 구분(검증됨/검증중/미검증)을 위해 상태별 건수가 필요하므로 미검증 **건수**(카운트)는 노출하되 **본문·식별정보는 절대 비노출**(0002 누설 금지와 양립 — 숫자만). 만약 미검증 카운트 노출도 민감하면 보수적 대안(검증됨만 집계)으로 축소 가능 — 본 스펙 기본은 상태별 건수 노출(디자인의 3색 범례 전제). 구현 착수 시 1줄 합의로 확정.
3. **sido null/미지 = "미지정" 버킷**으로 분리 표기(제외 아님 — 누락 합계 방지). 좌표 상수에 없는 sido 문자열도 미지정으로 폴백.
4. **조회수 = 공개 상세 조회 시 단순 +1**(유니크·봇 필터 없음, MVP). report 에 `viewCount`(integer, default 0) 컬럼 추가. 증가는 원자적 UPDATE(`viewCount = viewCount + 1`)로 경합 안전. verified 가 아니어서 404 되는 경로는 증가 전에 분기(미증가).
5. **map-stats 는 verified 범위 집계가 기본**, 상태별 색을 위해 reviewing/unverified 카운트가 필요하면 admin 가시성 vs 공개 가시성을 분리(공개는 결정 2 범위). 미검증 본문 누설 0 유지.

### 모델 / 마이그레이션
- report 에 `viewCount integer NOT NULL default 0` 추가, 새 마이그레이션(기존 불변). 백필 0.
- 지도 좌표는 **데이터가 아닌 코드 상수** — 마이그레이션 불필요.

### 데이터 무결성
- viewCount 는 분석용 카운터(무결성 핵심 아님). report 수정 이력(0001)에는 포함되나, 조회수 증가만으로 report_history 를 폭증시키지 않도록 viewCount 증가는 **이력 스냅샷 대상에서 제외**(또는 별도 경량 업데이트 경로). 결정: viewCount 는 history append 트리거 밖에서 원자 증가(감사 노이즈 회피).

### 회귀 (절대 조건)
- 기존 0001~0012 테스트 전부 통과. 공개 상세에 viewCount 추가는 직렬화 확장일 뿐 민감 필드 비노출·verified-only 게이트 불변. 0005 목록/필터 동작 불변.
